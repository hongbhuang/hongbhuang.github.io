# Project Level Rules

## CI/CD Workflow Routine (Automated Deployment)

This document defines the standard operating procedure (SOP) for finalizing tasks in this project to ensure every completed task is verified and deployed automatically.

### Trigger Condition
- The user signals task completion (e.g., "搞定", "完成任务", or equivalent).

### Workflow Steps

#### 1. Build Verification
Run `npm run build` within the project root.
- **Success Criterion:** Exit code `0`.
- **Failure Action:** Halt workflow, report specific error logs to user, do NOT commit/push.

#### 2. Code Cleanup & Audit
- Ensure no `console.log` or debug statements used for troubleshooting remain in the codebase.
- Verify that all files are staged correctly and no temporary system files (like `.log` or temp assets) are being included.

#### 3. Semantic Commit
Perform a single atomic commit using [Conventional Commits](https://www.conventionalcommits.org/):
- `feat: <description>` - New features.
- `fix: <description>` - Bug fixes.
- `style: <description>` - Changes that do not affect the meaning of the code (whitespace, formatting, etc.).
- `refactor: <description>` - A code change that neither fixes a bug nor adds a feature.
- `docs: <description>` - Documentation only changes.

#### 4. Remote Push
Execute `git push origin <current_branch>`.

### Failure Handling
- If any step fails, the agent must present a summary of the failure to the user and wait for manual intervention or a fix. Do not attempt multiple automatic retries on build failures unless explicitly instructed.
